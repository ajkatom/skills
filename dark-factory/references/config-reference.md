# dark-factory config.json — schema v0 (M1)

Lives at `<control_root>/config.json`. JSON, not YAML: runtime is stdlib-only
(divergence from the spec sketch, which shows YAML; a YAML front end can come later).

| Field | Type | M1 rule |
|---|---|---|
| `config_version` | str | `"0.1"` |
| `autonomy` | int | `4` (default, absent ok) or `5`. **L5 gate (spec 2.2):** `autonomy: 5` (lights-off) requires `assurance: "hardened"`, else `ConfigError`. Any other value (including bool or string) is rejected. |
| `checkpoint` | str | `"pause"` \| `"auto"`. Default: `pause` at autonomy 4, `auto` at autonomy 5 (using the resolved/defaulted autonomy value, so an absent `autonomy` still defaults checkpoint to `pause`). `pause` stops the loop after each non-converging iteration (exit 10) for human review via `resume`. |
| `assurance` | str | `cooperative` (unqualified, honor-system), `standard` (probe-verified OS read-denial → qualified), or `hardened` (builder runs in a Docker container, control root never mounted → qualified). `standard` requires a platform sandbox backend + a passing startup denial probe; `hardened` additionally requires a running Docker daemon + a passing container probe. Either fails closed (or downgrades with `--allow-downgrade`: hardened → standard if the OS sandbox is still healthy, else → cooperative). Other tier names are rejected. |
| `hardened.image` | str | optional, default `python:3.12-alpine` (`df_container.DEFAULT_IMAGE`). Non-empty; must not look like a CLI flag. Real cross-model builders need a user-supplied image with the CLI + credentials baked in. |
| `hardened.network` | str | optional, default `"none"`. `"none"` \| `"bridge"`. `"bridge"` is unrestricted egress (needed for real builder CLIs' API calls) and is recorded on the manifest so a reader can see the residual channel; provider-only egress enforcement is deferred (M12). |
| `hardened.memory` | str | optional, default `"2g"`. Must match `^[0-9]+[bkmg]$` (lowercase only, e.g. `"2g"`, `"512m"`). Passed to `docker run --memory`. |
| `hardened.pids` | int | optional, default `256`. Must be `>= 16` (bool rejected). Passed to `docker run --pids-limit`. |
| `hardened.dep_cache_dir` | str | optional (M26, spec §7.3), default `null`/unset. Absolute path to an existing, operator-provisioned directory (see `scripts/df_depcache.py`, `references/hardened.md` "Pinned dependency cache"). When set, resolved to a realpath at config-load time (must exist, must be a directory, else `ConfigError` naming `dep_cache_dir`); the supervisor bind-mounts it read-only into the hardened/enterprise builder container (a second `ro_mount`, re-checked disjoint from the control root right before mounting — same TOCTOU discipline as `roles.builder.adapter`'s directory) and sets `PIP_NO_INDEX=1`, `PIP_FIND_LINKS=<dir>/pypi`, `npm_config_cache=<dir>/npm-cache`, `npm_config_offline=true` in the container env. Unset → byte-identical to pre-M26 behavior: no extra mount, no env vars. Also read at `assurance: "enterprise"` (the same `hardened` block configures both tiers' container). |
| `candidate_network` | str | optional (M27, spec §7.4), default `"unrestricted"`. `"unrestricted"` \| `"deny"` \| `"loopback"`. Restricts the network authority of the CANDIDATE (the built app) only — applied to the verifier/candidate exec wrapper, never the builder, which keeps whatever network its tier already grants for provider API calls. `"deny"` blocks all network I/O including loopback; `"loopback"` blocks everything except `127.0.0.1` (macOS only — see `references/isolation.md`'s Linux-loopback limit). Non-`"unrestricted"` requires `assurance: standard` or above, else `ConfigError` (`"cooperative has no sandbox to enforce it"`); `"deny"` with `twins.enabled: true` is also a `ConfigError` (`"would make configured twins unreachable; use 'loopback' (macOS) or remove twins"`); `"deny"` with any http-type scenario is refused at a separate `supervisor.py` pre-build gate (scenarios aren't loaded at config time), naming every offending scenario id. Whichever mode is requested, `resolve_candidate_prefix` live-probes it via `df_sandbox.probe_network_denial` before the candidate ever runs under it — a failing probe fails the run closed (`PROBE_FAILED`, exit 2), never a silent unrestricted fallback. See `references/isolation.md`'s "Candidate network authority (§7.4)" section for the full model and honest limits (macOS DNS/mDNSResponder side channel, Linux `loopback` unsupported). |
| — | | A `hardened` block present while `assurance != "hardened"` is a `ConfigError` ("hardened block requires assurance: hardened"). At any tier, `cfg["_container"]` is always populated (with defaults when the block is absent); only consulted by the supervisor at effective tier `hardened`. |
| — | | **hardened ⇒ signed audit (spec 7):** at `assurance: "hardened"`, `audit.signing` defaults to `true` (absent means "on"); an explicit `audit.signing: false` is a `ConfigError` ("hardened requires signed audit manifests"). |
| `feedback` | str | must be `"ids"` in M1 |
| `max_iterations` | int | 1..20 |
| `workspace_root` | str | absolute path; must be disjoint from the control root |
| `roles.builder.adapter` | str | path to a protocol-0.1 adapter executable. Shipped: `scripts/adapters/{claude,codex,gemini,api_anthropic,api_openai}` (the last two are stdlib HTTP clients, no CLI required — see `references/role-adapters.md`). The chosen model's CLI must be installed (no silent fallback — an absent CLI aborts the run). **At `assurance: "hardened"`** it must additionally be an absolute path to an existing file whose directory is disjoint from the control root — that directory is bind-mounted ro into the builder container, so a bare command name (resolves against CWD) or a control-root-resident adapter would breach the holdout barrier and is a `ConfigError`; the supervisor re-verifies disjointness right before every mount (defense in depth). |
| `roles.builder.timeout_s` | int | optional, default 600 |
| `budget.billing` | str | `"api"` \| `"subscription"`. Default `"subscription"` (alert-only — dollars can't be metered). `"api"` enforces `budget.max_usd` via the per-call estimate. |
| `budget.max_usd` | float | optional; must be > 0. Dollar cap; enforced only when `billing: "api"` and `budget.per_call_usd` is also set. If `max_usd` is set without `per_call_usd`, the cap is recorded but downgraded to alert-only (no authoritative usage estimate). |
| `budget.per_call_usd` | float | optional; must be > 0. Estimated dollar cost reserved per builder call (admission control), not metered usage. |
| `budget.max_calls` | int | optional; must be >= 1. Exact hard cap on builder calls, enforced under any billing mode. |
| `budget.alert_at` | float | default `0.85`. Fraction of the cap (0, 1] at which a `BUDGET_ALERT` fires (warn, continue) before the 100% pause. |
| `budget.notification_sink` | str | optional, default `""` (no delivery). Destination for `BUDGET_ALERT`/`BUDGET_PAUSE` events, delivered via `df_notify.deliver()` (M18): `http://` \| `https://` (POST json) or `file:///abs/path` (append ndjson, no host component). Validated at config-load time — anything else is a `ConfigError`. Delivery is fail-**soft** (a failure journals `NOTIFY_FAILED` and never changes the run's exit code/outcome — contrast the fail-closed audit sink) and best-effort/fire-and-forget (single attempt, `timeout_s=10`, no retry queue). The event is redacted (M11 `redactor.redact_obj`) before it leaves the process. |
| `budget.notification_durable` | bool | optional, default `false` (M22). `true` upgrades notification delivery to **at-least-once**: `df_notify.deliver_durable()` retries `budget.notification_attempts` times, and on final failure spools the (redacted) event to `<control_root>/.notify-spool/pending.ndjson` instead of dropping it (journals `NOTIFY_SPOOLED`, not `NOTIFY_FAILED`); the supervisor flushes the spool at every run start (journals `NOTIFY_FLUSH`). STILL fail-soft — never changes the run's exit code/outcome. Local disk spool only, not a cross-host queue (see `references/budget.md`). Absent → M18 best-effort path, byte-identical. |
| `budget.notification_attempts` | int | optional, default `3` (M22). Must be `>= 1` (bool rejected). Number of `deliver()` attempts `deliver_durable()` makes before spooling. Only consulted when `budget.notification_durable: true`. |
| `budget.token_pricing` | dict | optional, default `{}` (M25). `{"<model>" \| "default": {"input_per_mtok": float >= 0, "output_per_mtok": float >= 0}}` — dollars per MILLION tokens. Each entry must be a JSON object with both fields present and numeric (bool rejected); a negative value, a non-dict entry, or a non-dict `token_pricing` block is a `ConfigError`. Absent → no pricing, `actual_usd` on the manifest stays `null`. Only adapters that report real usage (`api_anthropic`, `api_openai`; `usage.known=true`) ever get a non-null `actual_usd`, and only the `"default"` entry is currently consulted (model-keyed entries are validated for forward-compat but not yet selected — see `references/budget.md`). Never used for admission — see `references/budget.md` ("Real metering"). |
| — | | **Manifest additive (M25):** `usage: {"known": bool, "input_tokens": int, "output_tokens": int, "actual_usd": float \| null}` on every terminal manifest, fresh or resumed — same "seeded early, present on every abort branch" pattern as `budget`. `known` is `true` only when the builder adapter actually reported usage this run (`api_anthropic`/`api_openai`; CLI adapters always `false`); `input_tokens`/`output_tokens` are the run-total authoritative counts (`0` when `known:false`); `actual_usd` is non-null only when `known:true` AND `budget.token_pricing.default` is configured — otherwise `null`. Never gates admission; `estimated_usd` (on `budget`) alone drives the M8 admission/alert/pause path, unchanged. See `references/budget.md`. |
| `knowledge_base.kind` | str | optional: `none` (default) \| `wiki` \| `open-brain`. Enables optional grounding + opt-in run-summary write-back. |
| `knowledge_base.path` | str | required existing directory when kind=`wiki`; the run summary is appended to `<path>/dark-factory-runs.md`. |
| `knowledge_base.write_back` | bool | default false. When true + kind=`wiki`, the supervisor appends a barrier-safe run summary (outcome/tier/qualified/iterations/failing behavior IDs — no scenario text). `open-brain` write-back is done by the Claude session (MCP), not the supervisor. |
| `twins.enabled` | bool | default false. When true, the supervisor launches the twin services defined in `<control_root>/twins/*.json` around build/verify and exposes each as `DF_TWIN_<NAME>`. Requires ≥1 twin def. |
| `twins.startup_timeout_s` | int | 1..120, default 20. Max seconds to wait for a twin to write its endpoint file before the run aborts (exit 2). |
| `audit.signing` | bool | default false. When true the supervisor HMAC-signs each run manifest with a supervisor-only key, writing a `manifest.hmac` sidecar; `verify-manifest` then checks the signature (tamper-evident while the key stays secret). |
| `audit.key_path` | str | default `~/.dark-factory/audit.key` (mode 0600). MUST be outside the control root and workspace. Never written into any run artifact. |
| `audit.sink.kind` | str | optional block (M13); default `"none"`. `"none"` \| `"http-append"` \| `"s3-objectlock"`. Absent block → `cfg["_audit"]["sink"] = {"kind": "none", "required": False}` (byte-identical to pre-M13 behavior — nothing is ever pushed). Off-box push target for each finalized hash-chain entry (`df_audit_sink.push`); see `references/audit.md` for the trust-domain discussion. |
| `audit.sink.required` | bool | default false. When true, a push failure at finalize aborts the run (fail-closed, `AUDIT_SINK_FAILED` in `audit_events.jsonl`, nonzero exit); the local chain file is still written either way. When false, a failure only records `AUDIT_SINK_WARN` (in `audit_events.jsonl`) and the run proceeds. |
| `audit.sink.url` | str | required for `kind: "http-append"`; must match `^https?://\S+$`. PUT target is `{url}/audit/{key}` against a `df_audit_receiver.py`-compatible append-only receiver (201/200 new, 409 duplicate — never overwritten). |
| `audit.sink.endpoint`, `.bucket`, `.region` | str | all three required for `kind: "s3-objectlock"` (non-empty strings), else `ConfigError`. SigV4-signed stdlib PUT to `{endpoint}/{bucket}/{prefix}{key}`; WORM is enforced by the bucket's server-side object-lock retention config, not by this client. |
| `audit.sink.prefix` | str | s3-objectlock only; default `""`. Prepended to the key in the object path. |
| `audit.sink.access_key_env`, `.secret_key_env` | str | s3-objectlock only; the NAME of an environment variable, must match `^[A-Z][A-Z0-9_]*$`. Defaults `DF_AUDIT_S3_ACCESS_KEY` / `DF_AUDIT_S3_SECRET_KEY`. `push()` resolves the actual credential from `os.environ` at call time — config.json never carries a credential value. |
| — | | **No inline secrets, ever:** an `audit.sink.secret_key` or `audit.sink.access_key` field (a literal value rather than an `*_env` name) is a `ConfigError` regardless of `kind` — the leak risk (config.json is read by every run and hashed into `_config_sha256`) doesn't depend on whether the sink is currently active. |
| — | | **Hash chain (M13, no config key — always on):** every run, regardless of `audit.*`, appends one entry to `<control_root>/audit-chain.jsonl` immediately after its manifest is finalized, and writes that same entry to `runs/<id>/audit_chain.json`. Check the whole control root with `supervisor.py verify-chain <control_root> [--key-path <keyfile>]` (`OK: N entries` / exit 0, or the first break / exit 1 — fail-closed without `--key-path` if any entry is signed). See `references/audit.md`. |
| — | | **Sidecars + a separate event log, not manifest or journal fields:** `audit_chain.json` and (with a sink configured) `audit_sink_receipt.json` live in `runs/<id>/`, never as `manifest.json` fields — the chain entry binds the manifest's own already-finalized digest, so it can't be embedded back into that same manifest. The audit-anchor events (`AUDIT_CHAINED` always; `AUDIT_SINK_OK`/`AUDIT_SINK_WARN`/`AUDIT_SINK_FAILED` with a sink) go to a separate `runs/<id>/audit_events.jsonl` — NOT `journal.jsonl`, which `finalize_manifest` has already sealed. `verify-manifest` still hashes `journal.jsonl` **whole** against `manifest.journal_sha256` (unchanged from M5a — no `journal_bytes`, no narrowing). The authoritative audit records are `audit-chain.jsonl` (checked by `verify-chain`) + the sidecars; `audit_events.jsonl` is a convenience trail only. |
| `security_gates.enabled` | bool | default false. When true, mandatory security gates run on the converged artifact (independent of scenario pass) after the final exam and before CONVERGED; a failure listed in `fail_on` rejects the run. |
| `security_gates.secret_scan` | bool | default true when enabled. Runs `df_security.secret_scan` (private keys, cloud/Slack tokens, generic secret assignments) over the workspace. |
| `security_gates.dangerous_scan` | bool | default true when enabled. Runs `df_security.dangerous_scan` (`eval`/`exec`, `os.system`, `shell=True`, `pickle.loads`, unsafe `yaml.load`) over `*.py` files. |
| `security_gates.sbom` | bool | default true when enabled. Runs `df_security.sbom` (declared-dependency inventory); always `pass` — informational unless referenced in `fail_on`, where it's still a no-op (no failure condition). |
| `security_gates.external` | list | default `[]`. Each `{"name": str, "cmd": [str, ...]}` is a pluggable external gate (e.g. `bandit`, `semgrep`, `trufflehog`). Missing command (`shutil.which` miss) or a spawn error/timeout → `unavailable`, never a silent pass. |
| `security_gates.license.enabled` | bool | optional sub-block (M18); default `false`. Absent block → `cfg["_security"]["license"] = {"enabled": False, "allowlist": [], "require_license": False}` — byte-identical to pre-M18 (no `license` gate ever runs). When `true`, runs `df_security.license_scan` (spec §7.6) over the workspace: a declared/vendored license not in `allowlist` → `"disallowed-license"`; with `require_license: true`, a discovered package with no declared license at all → `"missing-license"`. See `references/security-gates.md` for the honest scope (declared/vendored metadata physically present in the artifact only — no network, no un-vendored transitive resolution). |
| `security_gates.license.allowlist` | list[str] | default `[]`. SPDX-style license identifiers, matched case-insensitively against every declared license found. An empty list disallows every declared license (the operator must list them explicitly). |
| `security_gates.license.require_license` | bool | default `false`. When true, a discovered package (root `pyproject.toml`/`package.json`, or a vendored `node_modules/*/package.json`/`*.dist-info/METADATA`) with no declared license value at all is a `"missing-license"` finding; when false, an undeclared license is silently not flagged (only mismatches against a declared value are). |
| `security_gates.dependency_audit.enabled` | bool | optional sub-block (M23); default `false`. Absent block → `cfg["_security"]["dependency_audit"] = {"enabled": False, "source": None, "snapshot_path": None, "ecosystems": [], "timeout_s": 20}` — byte-identical to pre-M23 (no `dependency_audit` gate ever runs, zero network calls). When `true`, runs `df_depaudit.parse_installed(workspace)` then queries OSV for known CVEs against each PINNED dependency, via whichever backend `source` selects. **TIER POLICY:** `source: "osv-api"` + `assurance` ∈ {`hardened`,`enterprise`} → `ConfigError` (see below); `source: "osv-snapshot"` is allowed at every tier. See `references/security-gates.md` for the full tier-policy table, the egress caveat, and the offline matcher's honest scope. |
| `security_gates.dependency_audit.source` | str | required when `enabled: true`: `"osv-api"` (live network query against `api.osv.dev` — egresses dependency names+versions; forbidden at `hardened`/`enterprise`) or `"osv-snapshot"` (a pre-provisioned local OSV export, queried fully offline — allowed at every tier). Anything else, or omitted while enabled, is a `ConfigError`. |
| `security_gates.dependency_audit.snapshot_path` | str | required (and must be an existing directory) when `source: "osv-snapshot"`, else `ConfigError`. Provisioned out-of-band by the operator via `df_depaudit.fetch_snapshot(ecosystem, dest_dir)` — not fetched during a run. Freshness is the operator's responsibility (see `references/security-gates.md`). |
| `security_gates.dependency_audit.ecosystems` | list[str] | optional, default `[]` (no filter — every ecosystem `parse_installed` finds, e.g. `PyPI`/`npm`, is checked). When non-empty, restricts the audit to only the listed ecosystems. |
| `security_gates.dependency_audit.timeout_s` | int | optional, default `20`. Must be in `1..300`. Per-package timeout passed to `query_osv_api` when `source: "osv-api"`; unused (no network) when `source: "osv-snapshot"`. |
| `security_gates.fail_on` | list[str] | default `["secret_scan", "dangerous_scan"]` when enabled. Gate names that are mandatory; must be built-ins (`secret_scan`,`dangerous_scan`,`sbom`,`license`,`dependency_audit`) or a declared `external[].name`, else `ConfigError`. A listed gate that's `fail` (or `unavailable` under `strict_unavailable`) fails the run. |
| `security_gates.strict_unavailable` | bool | default true. When true, a `fail_on` gate that comes back `unavailable` (external tool missing/errored) counts as a failure — fail-closed: a mandatory gate you can't run is not a pass. |
| _mandatory at standard+ (M33a)_ | — | At `assurance` ∈ {`standard`,`hardened`,`enterprise`}, `secret_scan` + `dangerous_scan` are FORCED enabled, unioned into `fail_on`, and `strict_unavailable` forced `true` — whether or not a `security_gates` block is present. An explicit `enabled: true` + a mandatory gate set `false` is a `ConfigError` (*"may strengthen security gates, never disable secret_scan"*). `cooperative` is unaffected (gates stay opt-in). |
| `security_gates.waivers.signers` | list[str] | optional sub-block (M33a); default `[]`. TIER-INDEPENDENT. ed25519 PUBLIC keys as 64-hex, lowercased + deduped, validated shape-only with the same stdlib `_HEX64_RE` `custody.approvers` uses (this module never imports `cryptography`). Governs which signers may sign a `df-waiver`; sealed into the run's manifest (`security.waiver_policy`). **A non-empty list REQUIRES `audit.signing: true`** (explicit `false` → `ConfigError`; absent/defaulted-false → forced on) so the sealed allowlist rides an HMAC-signed manifest and can't be widened post-run. |
| `security_gates.waivers.threshold` | int | default `0` (absent block → `{"signers": [], "threshold": 0}`, the fail-closed default — threshold 0 is never satisfiable, so no waiver is ever accepted). When signers are present, must be `1..len(signers)`; a positive threshold with no signers is a `ConfigError`. K distinct allowlisted signers must sign a finding for it to be waived. See `references/security-gates.md`. |
| `credentials.source` | str | optional block; absent → `cfg["_credentials"] = None` (no behavior change: exactly today's env at every tier). Required when the block is present: `"env-file"` \| `"keychain"` \| `"env"` (the launcher's own environment). Anything else is a `ConfigError`. |
| `credentials.env_file` | str | required iff `source: "env-file"`; `~` expanded, then must be absolute — a relative path is a `ConfigError`. Must be disjoint from BOTH the control root and `workspace_root` (a run must never be able to read its own credential file). Resolution (existence, permissions ≤ `0600`, `.gitignore`/`git check-ignore` verification) happens at run start via `df_creds.load_credentials`, never at config load. |
| `credentials.service_prefix` | str | keychain only; default `"dark-factory/"`. The macOS `security` lookup service name is `f"{service_prefix}{NAME}"` for each allowlisted `NAME`. |
| `credentials.allowlist` | list[str] | required, non-empty. Each entry must match `^[A-Z][A-Z0-9_]*$` and the list must not contain duplicates, else `ConfigError`. This is the ONLY set of variable names the builder can ever receive — nothing outside it is ever brokered, and at `hardened` nothing outside it enters the container env (`HOME=/tmp` from `build_argv` is the sole, already-documented exception). |
| — | | **Fail-closed resolution (spec, M11):** the supervisor resolves `credentials.allowlist` via `df_creds.load_credentials` BEFORE any builder call, in both a fresh `run` and every `resume` (credentials are re-resolved on resume, never cached in `state.json`). A `CredsError` (missing env-file, permission mode, ungitignored env-file inside a git work tree, missing keychain item, unset launcher var, empty value) is a refusal: `dark-factory: credentials: <detail>` to stderr, exit 2, nothing written to `run_dir`. |
| — | | **Builder env brokering:** at `hardened`, resolved values reach the container ONLY as `-e NAME=value` argv baked into the `docker run` invocation by `df_container.build_argv` — never via the docker client process's own env (residual: `-e` argv is visible to a local `ps`; see `references/credentials.md`). At `standard`/`cooperative`, the builder's env is `os.environ` with every var matching `(_API_KEY\|_TOKEN\|_SECRET\|_PASSWORD)$` that is NOT in `allowlist` stripped (`df_creds.launcher_scoped_env`), then the resolved credentials merged in last — a full env replacement, not a merge-over-inherit. |
| — | | **Redacted artifacts:** every persisted run artifact that could carry a credential value (journal, manifest, `state.json`, checkpoint reports, `verifier_report_iter_*.json`, `final_exam_report.json`, `security_report.json`) is passed through a `df_creds.Redactor` built from the resolved values before serialization; absent credentials block → the redactor is `None`, a strict no-op (zero behavior change). The additive manifest field `credentials: {"source": ..., "allowlist": [...]}` (or `null`) is names/allowlist only — it NEVER carries a value — on every terminal manifest, fresh or resumed. |
| `brownfield.mode` | str | optional, default `"auto"`. `"auto"` \| `"greenfield"` \| `"brownfield"`, else `ConfigError`. **Fail-safe toward brownfield (spec §8):** under `auto`, the supervisor classifies the run as `brownfield` iff `--project-src` is given AND its snapshot has ≥1 file — a non-empty existing tree is never silently treated as greenfield. Explicit `"brownfield"` requires a non-empty `project_src`, else a runtime refusal (`dark-factory: brownfield: ...`, exit 2) — there would be nothing to characterize. Explicit `"greenfield"` always skips characterization even with a non-empty `project_src`; the manifest then records `characterization.legacy_ignored: true` (an honest record of the human's override). |
| `brownfield.probes` | list | optional, default `[]`. Each `{"id": str, "run": [str, ...], "timeout_s": int}` is a human-curated probe the supervisor runs against a throwaway copy of `project_src` to freeze its CURRENT observed behavior into a holdout dev-cohort regression scenario (`BHV-REGRESS-<n>`) BEFORE the builder ever touches anything. `id` must match `^[a-z0-9-]{1,32}$` and be unique; `run` must be a non-empty list of strings; `timeout_s` must be an int in `1..120`. `mode: "brownfield"` with an empty `probes` list is a `ConfigError` at load time (a brownfield run with nothing to characterize would falsely claim regression coverage); a non-empty `probes` with `mode: "greenfield"` is also a `ConfigError` (contradictory: greenfield explicitly skips characterization). An **auto-detected** brownfield run (mode `auto`, non-empty `project_src`, but zero probes configured — e.g. every pre-M15 project-src run) is a back-compat no-op: the manifest still honestly reports `mode: "brownfield"`, but characterization is skipped (`generated: 0`) rather than erroring, since `df_config` only forbids the empty-probes combination when `mode` is the *explicit* `"brownfield"` value. |
| — | | **When characterization runs:** after isolation is resolved (probes execute under the same `exec_wrapper`/sandbox the verifier uses — a probe can read the snapshot copy but the control root stays denied, same as any scenario run) and BEFORE the first build iteration. Each generated scenario is written to `<run_dir>/generated-scenarios/<id>.json` (control-plane, alongside `scenarios/` — never inside the builder's workspace) and journaled as `CHARACTERIZED(mode, generated, behavior_ids)`; `MODE_DETECTED(mode, legacy_ignored)` is always journaled, brownfield or not. The generated set is unioned into the verifier's DEV-cohort scenario load ONLY (`run_scenarios.load_scenarios`'s `extra_scenarios_dir`) — a scenario `id` colliding between `scenarios/` and `generated-scenarios/` is an `OracleError`. The barrier holds exactly like every other scenario: generated content is NEVER copied into the workspace, the prompt, or feedback — only behavior IDs + taxonomy cross back, so a builder that breaks a captured behavior sees `BHV-REGRESS-<n>` fail with no indication of what the probe actually checked. |
