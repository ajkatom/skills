# dark-factory config.json ‚ÄĒ schema v0 (M1)

Lives at `<control_root>/config.json`. JSON, not YAML: runtime is stdlib-only
(divergence from the spec sketch, which shows YAML; a YAML front end can come later).

| Field | Type | M1 rule |
|---|---|---|
| `config_version` | str | `"0.1"` |
| `autonomy` | int | `4` (default, absent ok) or `5`. **L5 gate (spec 2.2):** `autonomy: 5` (lights-off) requires `assurance: "hardened"`, else `ConfigError`. Any other value (including bool or string) is rejected. |
| `checkpoint` | str | `"pause"` \| `"auto"`. Default: `pause` at autonomy 4, `auto` at autonomy 5 (using the resolved/defaulted autonomy value, so an absent `autonomy` still defaults checkpoint to `pause`). `pause` stops the loop after each non-converging iteration (exit 10) for human review via `resume`. |
| `assurance` | str | `cooperative` (unqualified, honor-system), `standard` (probe-verified OS read-denial ‚Üí qualified), or `hardened` (builder runs in a Docker container, control root never mounted ‚Üí qualified). `standard` requires a platform sandbox backend + a passing startup denial probe; `hardened` additionally requires a running Docker daemon + a passing container probe. Either fails closed (or downgrades with `--allow-downgrade`: hardened ‚Üí standard if the OS sandbox is still healthy, else ‚Üí cooperative). Other tier names are rejected. |
| `hardened.image` | str | optional, default `python:3.12-alpine` (`df_container.DEFAULT_IMAGE`). Non-empty; must not look like a CLI flag. Real cross-model builders need a user-supplied image with the CLI + credentials baked in. |
| `hardened.network` | str | optional, default `"none"`. `"none"` \| `"bridge"`. `"bridge"` is unrestricted egress (needed for real builder CLIs' API calls) and is recorded on the manifest so a reader can see the residual channel; provider-only egress enforcement is deferred (M12). |
| `hardened.memory` | str | optional, default `"2g"`. Must match `^[0-9]+[bkmg]$` (lowercase only, e.g. `"2g"`, `"512m"`). Passed to `docker run --memory`. |
| `hardened.pids` | int | optional, default `256`. Must be `>= 16` (bool rejected). Passed to `docker run --pids-limit`. |
| ‚ÄĒ | | A `hardened` block present while `assurance != "hardened"` is a `ConfigError` ("hardened block requires assurance: hardened"). At any tier, `cfg["_container"]` is always populated (with defaults when the block is absent); only consulted by the supervisor at effective tier `hardened`. |
| ‚ÄĒ | | **hardened ‚áí signed audit (spec 7):** at `assurance: "hardened"`, `audit.signing` defaults to `true` (absent means "on"); an explicit `audit.signing: false` is a `ConfigError` ("hardened requires signed audit manifests"). |
| `feedback` | str | must be `"ids"` in M1 |
| `max_iterations` | int | 1..20 |
| `workspace_root` | str | absolute path; must be disjoint from the control root |
| `roles.builder.adapter` | str | path to a protocol-0.1 adapter executable. Shipped: `scripts/adapters/{claude,codex,gemini}`. The chosen model's CLI must be installed (no silent fallback ‚ÄĒ an absent CLI aborts the run). **At `assurance: "hardened"`** it must additionally be an absolute path to an existing file whose directory is disjoint from the control root ‚ÄĒ that directory is bind-mounted ro into the builder container, so a bare command name (resolves against CWD) or a control-root-resident adapter would breach the holdout barrier and is a `ConfigError`; the supervisor re-verifies disjointness right before every mount (defense in depth). |
| `roles.builder.timeout_s` | int | optional, default 600 |
| `budget.billing` | str | `"api"` \| `"subscription"`. Default `"subscription"` (alert-only ‚ÄĒ dollars can't be metered). `"api"` enforces `budget.max_usd` via the per-call estimate. |
| `budget.max_usd` | float | optional; must be > 0. Dollar cap; enforced only when `billing: "api"` and `budget.per_call_usd` is also set. If `max_usd` is set without `per_call_usd`, the cap is recorded but downgraded to alert-only (no authoritative usage estimate). |
| `budget.per_call_usd` | float | optional; must be > 0. Estimated dollar cost reserved per builder call (admission control), not metered usage. |
| `budget.max_calls` | int | optional; must be >= 1. Exact hard cap on builder calls, enforced under any billing mode. |
| `budget.alert_at` | float | default `0.85`. Fraction of the cap (0, 1] at which a `BUDGET_ALERT` fires (warn, continue) before the 100% pause. |
| `budget.notification_sink` | str | optional, default `""`. Recorded destination for L5 budget alerts; delivery is stubbed (journaled + printed) in M8. |
| `knowledge_base.kind` | str | optional: `none` (default) \| `wiki` \| `open-brain`. Enables optional grounding + opt-in run-summary write-back. |
| `knowledge_base.path` | str | required existing directory when kind=`wiki`; the run summary is appended to `<path>/dark-factory-runs.md`. |
| `knowledge_base.write_back` | bool | default false. When true + kind=`wiki`, the supervisor appends a barrier-safe run summary (outcome/tier/qualified/iterations/failing behavior IDs ‚ÄĒ no scenario text). `open-brain` write-back is done by the Claude session (MCP), not the supervisor. |
| `twins.enabled` | bool | default false. When true, the supervisor launches the twin services defined in `<control_root>/twins/*.json` around build/verify and exposes each as `DF_TWIN_<NAME>`. Requires ‚Č•1 twin def. |
| `twins.startup_timeout_s` | int | 1..120, default 20. Max seconds to wait for a twin to write its endpoint file before the run aborts (exit 2). |
| `audit.signing` | bool | default false. When true the supervisor HMAC-signs each run manifest with a supervisor-only key, writing a `manifest.hmac` sidecar; `verify-manifest` then checks the signature (tamper-evident while the key stays secret). |
| `audit.key_path` | str | default `~/.dark-factory/audit.key` (mode 0600). MUST be outside the control root and workspace. Never written into any run artifact. |
| `audit.sink.kind` | str | optional block (M13); default `"none"`. `"none"` \| `"http-append"` \| `"s3-objectlock"`. Absent block ‚Üí `cfg["_audit"]["sink"] = {"kind": "none", "required": False}` (byte-identical to pre-M13 behavior ‚ÄĒ nothing is ever pushed). Off-box push target for each finalized hash-chain entry (`df_audit_sink.push`); see `references/audit.md` for the trust-domain discussion. |
| `audit.sink.required` | bool | default false. When true, a push failure at finalize aborts the run (fail-closed, `AUDIT_SINK_FAILED`, nonzero exit); the local chain file is still written either way. When false, a failure only journals `AUDIT_SINK_WARN` and the run proceeds. |
| `audit.sink.url` | str | required for `kind: "http-append"`; must match `^https?://\S+$`. PUT target is `{url}/audit/{key}` against a `df_audit_receiver.py`-compatible append-only receiver (201/200 new, 409 duplicate ‚ÄĒ never overwritten). |
| `audit.sink.endpoint`, `.bucket`, `.region` | str | all three required for `kind: "s3-objectlock"` (non-empty strings), else `ConfigError`. SigV4-signed stdlib PUT to `{endpoint}/{bucket}/{prefix}{key}`; WORM is enforced by the bucket's server-side object-lock retention config, not by this client. |
| `audit.sink.prefix` | str | s3-objectlock only; default `""`. Prepended to the key in the object path. |
| `audit.sink.access_key_env`, `.secret_key_env` | str | s3-objectlock only; the NAME of an environment variable, must match `^[A-Z][A-Z0-9_]*$`. Defaults `DF_AUDIT_S3_ACCESS_KEY` / `DF_AUDIT_S3_SECRET_KEY`. `push()` resolves the actual credential from `os.environ` at call time ‚ÄĒ config.json never carries a credential value. |
| ‚ÄĒ | | **No inline secrets, ever:** an `audit.sink.secret_key` or `audit.sink.access_key` field (a literal value rather than an `*_env` name) is a `ConfigError` regardless of `kind` ‚ÄĒ the leak risk (config.json is read by every run and hashed into `_config_sha256`) doesn't depend on whether the sink is currently active. |
| `security_gates.enabled` | bool | default false. When true, mandatory security gates run on the converged artifact (independent of scenario pass) after the final exam and before CONVERGED; a failure listed in `fail_on` rejects the run. |
| `security_gates.secret_scan` | bool | default true when enabled. Runs `df_security.secret_scan` (private keys, cloud/Slack tokens, generic secret assignments) over the workspace. |
| `security_gates.dangerous_scan` | bool | default true when enabled. Runs `df_security.dangerous_scan` (`eval`/`exec`, `os.system`, `shell=True`, `pickle.loads`, unsafe `yaml.load`) over `*.py` files. |
| `security_gates.sbom` | bool | default true when enabled. Runs `df_security.sbom` (declared-dependency inventory); always `pass` ‚ÄĒ informational unless referenced in `fail_on`, where it's still a no-op (no failure condition). |
| `security_gates.external` | list | default `[]`. Each `{"name": str, "cmd": [str, ...]}` is a pluggable external gate (e.g. `bandit`, `semgrep`, `trufflehog`). Missing command (`shutil.which` miss) or a spawn error/timeout ‚Üí `unavailable`, never a silent pass. |
| `security_gates.fail_on` | list[str] | default `["secret_scan", "dangerous_scan"]` when enabled. Gate names that are mandatory; must be built-ins (`secret_scan`,`dangerous_scan`,`sbom`) or a declared `external[].name`, else `ConfigError`. A listed gate that's `fail` (or `unavailable` under `strict_unavailable`) fails the run. |
| `security_gates.strict_unavailable` | bool | default true. When true, a `fail_on` gate that comes back `unavailable` (external tool missing/errored) counts as a failure ‚ÄĒ fail-closed: a mandatory gate you can't run is not a pass. |
| `credentials.source` | str | optional block; absent ‚Üí `cfg["_credentials"] = None` (no behavior change: exactly today's env at every tier). Required when the block is present: `"env-file"` \| `"keychain"` \| `"env"` (the launcher's own environment). Anything else is a `ConfigError`. |
| `credentials.env_file` | str | required iff `source: "env-file"`; `~` expanded, then must be absolute ‚ÄĒ a relative path is a `ConfigError`. Must be disjoint from BOTH the control root and `workspace_root` (a run must never be able to read its own credential file). Resolution (existence, permissions ‚Č§ `0600`, `.gitignore`/`git check-ignore` verification) happens at run start via `df_creds.load_credentials`, never at config load. |
| `credentials.service_prefix` | str | keychain only; default `"dark-factory/"`. The macOS `security` lookup service name is `f"{service_prefix}{NAME}"` for each allowlisted `NAME`. |
| `credentials.allowlist` | list[str] | required, non-empty. Each entry must match `^[A-Z][A-Z0-9_]*$` and the list must not contain duplicates, else `ConfigError`. This is the ONLY set of variable names the builder can ever receive ‚ÄĒ nothing outside it is ever brokered, and at `hardened` nothing outside it enters the container env (`HOME=/tmp` from `build_argv` is the sole, already-documented exception). |
| ‚ÄĒ | | **Fail-closed resolution (spec, M11):** the supervisor resolves `credentials.allowlist` via `df_creds.load_credentials` BEFORE any builder call, in both a fresh `run` and every `resume` (credentials are re-resolved on resume, never cached in `state.json`). A `CredsError` (missing env-file, permission mode, ungitignored env-file inside a git work tree, missing keychain item, unset launcher var, empty value) is a refusal: `dark-factory: credentials: <detail>` to stderr, exit 2, nothing written to `run_dir`. |
| ‚ÄĒ | | **Builder env brokering:** at `hardened`, resolved values reach the container ONLY as `-e NAME=value` argv baked into the `docker run` invocation by `df_container.build_argv` ‚ÄĒ never via the docker client process's own env (residual: `-e` argv is visible to a local `ps`; see `references/credentials.md`). At `standard`/`cooperative`, the builder's env is `os.environ` with every var matching `(_API_KEY\|_TOKEN\|_SECRET\|_PASSWORD)$` that is NOT in `allowlist` stripped (`df_creds.launcher_scoped_env`), then the resolved credentials merged in last ‚ÄĒ a full env replacement, not a merge-over-inherit. |
| ‚ÄĒ | | **Redacted artifacts:** every persisted run artifact that could carry a credential value (journal, manifest, `state.json`, checkpoint reports, `verifier_report_iter_*.json`, `final_exam_report.json`, `security_report.json`) is passed through a `df_creds.Redactor` built from the resolved values before serialization; absent credentials block ‚Üí the redactor is `None`, a strict no-op (zero behavior change). The additive manifest field `credentials: {"source": ..., "allowlist": [...]}` (or `null`) is names/allowlist only ‚ÄĒ it NEVER carries a value ‚ÄĒ on every terminal manifest, fresh or resumed. |
| `brownfield.mode` | str | optional, default `"auto"`. `"auto"` \| `"greenfield"` \| `"brownfield"`, else `ConfigError`. **Fail-safe toward brownfield (spec ¬ß8):** under `auto`, the supervisor classifies the run as `brownfield` iff `--project-src` is given AND its snapshot has ‚Č•1 file ‚ÄĒ a non-empty existing tree is never silently treated as greenfield. Explicit `"brownfield"` requires a non-empty `project_src`, else a runtime refusal (`dark-factory: brownfield: ...`, exit 2) ‚ÄĒ there would be nothing to characterize. Explicit `"greenfield"` always skips characterization even with a non-empty `project_src`; the manifest then records `characterization.legacy_ignored: true` (an honest record of the human's override). |
| `brownfield.probes` | list | optional, default `[]`. Each `{"id": str, "run": [str, ...], "timeout_s": int}` is a human-curated probe the supervisor runs against a throwaway copy of `project_src` to freeze its CURRENT observed behavior into a holdout dev-cohort regression scenario (`BHV-REGRESS-<n>`) BEFORE the builder ever touches anything. `id` must match `^[a-z0-9-]{1,32}$` and be unique; `run` must be a non-empty list of strings; `timeout_s` must be an int in `1..120`. `mode: "brownfield"` with an empty `probes` list is a `ConfigError` at load time (a brownfield run with nothing to characterize would falsely claim regression coverage); a non-empty `probes` with `mode: "greenfield"` is also a `ConfigError` (contradictory: greenfield explicitly skips characterization). An **auto-detected** brownfield run (mode `auto`, non-empty `project_src`, but zero probes configured ‚ÄĒ e.g. every pre-M15 project-src run) is a back-compat no-op: the manifest still honestly reports `mode: "brownfield"`, but characterization is skipped (`generated: 0`) rather than erroring, since `df_config` only forbids the empty-probes combination when `mode` is the *explicit* `"brownfield"` value. |
| ‚ÄĒ | | **When characterization runs:** after isolation is resolved (probes execute under the same `exec_wrapper`/sandbox the verifier uses ‚ÄĒ a probe can read the snapshot copy but the control root stays denied, same as any scenario run) and BEFORE the first build iteration. Each generated scenario is written to `<run_dir>/generated-scenarios/<id>.json` (control-plane, alongside `scenarios/` ‚ÄĒ never inside the builder's workspace) and journaled as `CHARACTERIZED(mode, generated, behavior_ids)`; `MODE_DETECTED(mode, legacy_ignored)` is always journaled, brownfield or not. The generated set is unioned into the verifier's DEV-cohort scenario load ONLY (`run_scenarios.load_scenarios`'s `extra_scenarios_dir`) ‚ÄĒ a scenario `id` colliding between `scenarios/` and `generated-scenarios/` is an `OracleError`. The barrier holds exactly like every other scenario: generated content is NEVER copied into the workspace, the prompt, or feedback ‚ÄĒ only behavior IDs + taxonomy cross back, so a builder that breaks a captured behavior sees `BHV-REGRESS-<n>` fail with no indication of what the probe actually checked. |
| ‚ÄĒ | | **Resume never re-characterizes:** `resume()` doesn't receive `--project-src` and must not re-observe a possibly-changed source ‚ÄĒ it reuses the ORIGINAL run's `<run_dir>/generated-scenarios/` verbatim (recovering `mode`/`legacy_ignored` from the `MODE_DETECTED` journal entry, and `probes`/`generated` counts from `cfg["_brownfield"]["probes"]` + the sealed directory's file count) rather than calling `df_brownfield.characterize` again. |
| ‚ÄĒ | | **Manifest additive:** `mode: "auto"\|"greenfield"\|"brownfield"` and `characterization: {"probes": int, "generated": int, "note": "behavioral snapshot at probe points; unprobed behavior may regress", "legacy_ignored": bool}` when `mode == "brownfield"` or `legacy_ignored`, else `{"probes": 0, "generated": 0}` ‚ÄĒ on every terminal manifest, fresh or resumed, exactly like `credentials`/`twin_evidence`. **Honest scope:** characterization guards only what the probes exercise; unprobed behavior can still regress silently (see `references/brownfield.md`). |
