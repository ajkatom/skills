# Dark Factory Security Glossary

This glossary consolidates the security terminology used by Dark Factory. It includes project-specific controls and the underlying security concepts.

## Core Dark Factory concepts

- **Dark factory** — A development loop in which a builder receives a specification, creates an implementation, and gets limited failure feedback until it passes independently held acceptance tests.
- **Builder** — The AI or executable that writes the candidate implementation. It is treated as potentially curious or hostile and should not see the hidden scenarios.
- **Candidate** — The program produced by the builder. It is separately restricted while acceptance scenarios execute because generated code may itself be unsafe.
- **Verifier** — The trusted component that executes hidden scenarios against the candidate and calculates pass/fail results.
- **Oracle** — Deterministic logic that decides whether an observed result satisfies a scenario. Unlike an LLM judge, it uses explicit assertions.
- **Control root / control plane** — The protected directory containing hidden scenarios, run state, manifests, keys or policies, and audit evidence. The builder must not access it.
- **Workspace / artifact plane** — The directory where the builder creates the candidate. It is intentionally writable but contains no hidden test material.
- **Barrier** — The information-security boundary separating the builder from scenarios, expected outputs, verifier internals, and detailed failures.
- **Holdout scenario** — A secret acceptance test derived from the specification but withheld from the builder.
- **Dev cohort** — Hidden scenarios used repeatedly during the repair loop. Only limited behavior-ID feedback returns to the builder.
- **Final cohort / sealed exam** — Scenarios run exactly once after development converges. A failure rejects the artifact and is never fed back for another repair attempt.
- **Behavior ID** — An identifier such as `BHV-001` representing a requirement. Feedback can identify the failed behavior without disclosing the scenario.
- **Failure taxonomy** — A small predefined category such as wrong output, timeout, or property violation. It provides useful feedback without revealing secret inputs or expected answers.
- **Teaching to the test** — When the builder learns test-specific answers instead of implementing the intended general behavior. Hidden scenarios, variants, and final cohorts reduce this risk.
- **Scenario leakage** — Any disclosure of hidden inputs, expected outputs, scenario files, or sufficiently detailed failure information to the builder.
- **Spec leak gate** — A pre-build check that rejects scenarios whose exact expected answers already appear in the builder-visible specification.
- **Digital twin** — A controlled mock of an external service used for deterministic testing without contacting the real service.
- **Twin evidence** — Evidence generated by the twin itself proving that the candidate actually contacted it instead of returning a hardcoded answer.
- **Variant seed** — A verifier-only value that makes a twin return a fresh, unpredictable response during each pass.

## Assurance and intervention

- **Assurance tier** — The strength of the technical security controls. It is separate from how much human intervention the run allows.
- **Cooperative tier** — Honor-system separation only. There is no OS-enforced barrier, so every result is explicitly unqualified.
- **Standard tier** — Uses a probe-verified operating-system sandbox to deny builder access to the control root and constrain candidate access.
- **Hardened tier** — Runs the builder in Docker without mounting the control root. It also requires the standard-tier sandbox for verification and forces signed auditing.
- **Enterprise tier** — Hardened plus seccomp, kernel-enforced egress restrictions, a host credential proxy, required off-box auditing, builder confinement, and K-of-N approval.
- **Qualified** — The run satisfied every required isolation, candidate-egress, host-isolation, final-exam, application-security, audit, and applicable custody condition.
- **Unqualified** — The candidate may work, but the run lacks the security evidence required for an approved shipping candidate.
- **Downgrade** — An explicit reduction to a weaker tier when a requested control cannot be established. It must be requested, journaled, and reflected in qualification.
- **H1 Directed** — Pauses frequently: after failures, before rebuilds, at budget limits, and before shipping.
- **H2 Supervised** — Pauses after each non-converged verification and before shipping.
- **H3 Guarded** — Runs through iterations automatically and pauses only at hard budget controls.
- **H4 Lights-out** — Never requests resumable human intervention. Human-required situations become terminal failures. Only hardened or enterprise may use it.

## Fail-safe behavior

- **Fail closed** — If a security control cannot be proved, the operation is refused. A scanner being unavailable is not treated as a passing scan.
- **Fail open** — Continuing when a security check is missing or cannot execute.
- **Strict unavailable** — A security gate that cannot run counts as a failure rather than being skipped.
- **Live probe** — A runtime experiment showing that a control works on the current host, kernel, image, and configuration.
- **Canary** — Known test data placed where access should be forbidden. A sandbox passes its probe only if the process cannot read or modify it.
- **Non-vacuous probe** — A probe that also proves the test process actually ran; otherwise no leak might simply mean the probe never started.
- **Superset invariant** — Tightening policy may cause previously passing runs to fail, but must never cause an unsafe run to newly qualify.

## Sandboxing and process isolation

- **Sandbox** — A restricted execution environment controlling filesystem, network, process, and operating-system access.
- **OS read-denial sandbox** — An operating-system policy preventing a process from reading protected paths.
- **`sandbox-exec`** — The macOS sandbox mechanism used to apply file, network, and IPC policies.
- **`bwrap` / Bubblewrap** — A Linux sandbox tool that builds restricted mount, PID, and network namespaces.
- **Container barrier** — Isolation created by running the builder in a Docker container whose visible filesystem excludes the control root.
- **Denial by construction** — Preventing access by making a resource absent instead of exposing it and relying on a deny rule.
- **Bind mount** — Making a host file or directory visible inside a container.
- **Read-only mount** — A mount the container can inspect but cannot modify, such as an adapter or dependency cache.
- **Read-only root filesystem** — A container setting that prevents writes to the normal container filesystem.
- **`tmpfs`** — A temporary memory-backed filesystem used for writable scratch space or to mask a protected host path.
- **Ephemeral container** — A container removed after execution, reducing persistent state between builder calls.
- **Mount namespace** — A Linux mechanism giving a process its own view of mounted filesystems.
- **PID namespace** — A separate process-ID environment that improves descendant-process containment and cleanup.
- **Network namespace** — A separate network stack that can eliminate routes, interfaces, or access to the host network.
- **Linux capabilities** — Individually separable privileges traditionally held by root, such as mounting filesystems or changing network policy.
- **`--cap-drop ALL`** — Removes all Linux capabilities from the container or sandboxed process.
- **`CAP_SYS_ADMIN`** — A powerful capability permitting mount and namespace operations, including possible sandbox escapes if retained.
- **`NET_ADMIN`** — A capability allowing modification of network or firewall rules. Enterprise drops it so the builder cannot undo egress restrictions.
- **No new privileges** — A Linux restriction preventing a process from gaining privileges through setuid binaries or file capabilities.
- **Seccomp** — A Linux kernel filter controlling which system calls a process may execute.
- **System call / syscall** — A low-level request from a program to the kernel, such as `mount`, `ptrace`, or `unshare`.
- **Seccomp profile** — A JSON policy telling Docker which syscalls to allow, deny, or trap.
- **`ptrace`** — A syscall used to inspect or control another process. Restricting it reduces debugging-based attacks and credential theft.
- **Resource limit** — A restriction on memory, process count, time, or another finite resource.
- **PID limit** — A maximum number of processes, helping contain fork bombs.
- **Fork bomb** — A program that repeatedly creates processes until system resources are exhausted.
- **Process-group teardown** — Terminating the candidate and its descendants as a group after a scenario completes or times out.
- **Process-group escape** — A child uses `setsid`, double-forking, or similar techniques to leave the process group and avoid best-effort cleanup.
- **Trusted computing base (TCB)** — Everything that must work correctly for the security claim to hold, including the supervisor, host OS, sandbox, Docker daemon, image, verifier, and keys.

## Host and network authority

- **Default deny** — Access starts forbidden and only specifically required resources are allowed.
- **Host isolation** — Preventing the candidate from reading SSH files, dotfiles, other repositories, keychains, cloud configuration, and unrelated host data.
- **Host-read opt-out** — An explicit choice to permit host reads. The run may operate but cannot claim normal host-isolation qualification.
- **Candidate environment allowlist** — A minimal list of environment variables the candidate may inherit, such as `PATH`, locale, and temporary-directory settings.
- **Denylist scrub** — A secondary filter removing sensitive variables such as tokens, credentials, proxy configuration, `SSH_AUTH_SOCK`, and cloud-provider variables.
- **Ambient credential** — A credential inherited automatically from the operator's shell or desktop session rather than explicitly authorized for the run.
- **Egress** — Outbound network traffic from the candidate or builder.
- **Unrestricted egress** — Permission to contact any reachable host. At standard or higher, unrestricted candidate egress is disqualifying.
- **Network deny** — Blocks all candidate networking, including loopback.
- **Loopback-only** — Allows local services such as twins on `127.0.0.1` while blocking external destinations.
- **Kernel-locked egress** — Enterprise firewall rules enforced below the builder process and protected by removing network-administration privileges.
- **Destination allowlist** — The small set of hosts the proxy is permitted to contact, usually only the selected model provider.
- **DNS side channel** — Data leakage through domain-name lookups even when ordinary outbound sockets are restricted.
- **Mach IPC** — macOS inter-process communication. Services such as DNS or Keychain may create indirect access channels unless denied by the sandbox profile.

## Credentials and cryptography

- **Credential broker** — Resolves approved secrets host-side and injects only specifically authorized variables into the correct process.
- **Credential proxy** — An enterprise host-side service that adds the provider token to requests so the raw token never enters the builder container.
- **Credential allowlist** — Exact environment-variable names the builder may receive; unlisted secrets are excluded.
- **Redaction** — Replacing secret values with text such as `***REDACTED***` before writing logs, manifests, or reports.
- **Credential rotation** — Replacing credentials periodically or after exposure. The ordinary credential broker contains credentials but does not rotate them.
- **Private key** — Secret cryptographic material used to sign. It must remain with its approver.
- **Public key** — Non-secret material used to verify signatures.
- **Ed25519** — A modern asymmetric digital-signature scheme used for custody, waiver, release, and override approvals.
- **HMAC-SHA256** — A symmetric message-authentication code used to prove that manifest bytes were produced by someone holding the shared audit key.
- **Digital signature** — Cryptographic evidence binding a signer to exact bytes. Modifying the bytes invalidates the signature.
- **Digest / cryptographic hash** — A fixed-size fingerprint of data. Dark Factory primarily uses SHA-256.
- **Nonce** — A unique value used once to prevent the same signed approval from being replayed.
- **Replay attack** — Reusing a previously valid signed message to authorize a new or repeated operation.
- **K-of-N approval** — At least K distinct authorized signers out of N must approve the same claim.
- **Split custody** — Qualifying an enterprise artifact requires K-of-N distinct approver signatures over the sealed manifest. With `threshold ≥ 2` no single operator can ship; a `threshold` of 1 is permitted (and then a single approver suffices). K-of-N proves distinct KEYS/SIGNATURES, not distinct human owners — separation of duties depends on the operator keeping the private keys with genuinely separate people.
- **Separation of duties** — Assigning building, verification, approval, and release authority to distinct roles to reduce unilateral mistakes or abuse.

## Integrity, evidence, and auditing

- **Manifest** — The sealed run summary containing configuration hashes, outcomes, qualification substates, security results, and artifact identity.
- **Canonical bytes** — A deterministic serialization of the manifest so all signers sign exactly the same byte sequence.
- **Journal** — An append-oriented record of state transitions, dispatches, checkpoints, failures, and decisions.
- **Attestation** — A separately signed statement that a sealed manifest satisfies a policy such as custody, waiver, or release approval.
- **Artifact sealing** — Freezing the candidate into an immutable-style object before the final exam and security gates.
- **Artifact binding** — Recording the frozen artifact's cryptographic identity in the manifest so the evidence refers to the exact bytes tested.
- **Content-addressed object store** — Storage where an object's identifier is derived from its contents, modes, names, and directory structure.
- **Object ID** — The cryptographic identity of a sealed artifact.
- **Sidecar** — Metadata stored beside an object, such as file paths, modes, hashes, and counts.
- **Fresh materialization** — Creating a new workspace from the sealed object for the final exam, gates, or shipping.
- **Tamper-evident** — Unauthorized modification can be detected later through hashes or signatures.
- **Tamper-proof** — Modification is technically prevented. Dark Factory's local evidence is generally tamper-evident, not tamper-proof against the same OS user.
- **Hash chain** — Each audit entry includes the preceding entry's hash, making removal, insertion, or reordering detectable.
- **Off-box audit sink** — A destination on a different machine or account that receives audit entries so a local attacker cannot silently rewrite both the run and its only evidence.
- **Trust domain** — A boundary controlled by different credentials, systems, or administrators. A receiver on the same laptop is not meaningfully separate.
- **WORM** — “Write once, read many”: retained evidence can be read but not overwritten or deleted during its retention period.
- **S3 Object Lock** — An S3 mechanism for retention-backed WORM storage.
- **Receipt** — Evidence that a particular attestation or audit object was successfully stored off-box, bound to the transmitted bytes.
- **Detection-grade security** — Misbehavior or tampering should leave evidence and cause later verification to fail.
- **Prevention-grade security** — The attacker is technically unable to perform the prohibited action in the first place.
- **Same-user attacker** — A hostile process running under the same OS account, which is difficult to prevent from rewriting locally owned files or reading accessible symmetric keys.

## Application and supply-chain gates

- **Security gate** — A deterministic security check run against the sealed, converged artifact before qualification.
- **Security floor** — A mandatory minimum that catches known classes of problems but does not prove the application is secure.
- **Secret scan** — Searches the artifact for likely API keys, private keys, tokens, passwords, or credential patterns.
- **Dangerous-pattern scan** — Searches for high-risk implementation patterns such as unsafe shell execution, insecure deserialization, or disabled verification.
- **SAST** — Static Application Security Testing: code analysis intended to find vulnerabilities without executing the application. Dark Factory's built-in regex gates are not a complete SAST engine.
- **External gate** — A configured scanner such as Semgrep or Bandit invoked through a defined command/result interface.
- **SBOM** — Software Bill of Materials: an inventory of dependencies and components included in the artifact.
- **Dependency audit** — Checks declared dependency versions against known vulnerability information.
- **OSV** — Open Source Vulnerabilities, a standardized vulnerability-data format and database ecosystem.
- **License gate** — Checks dependency licenses against an allowed or denied policy.
- **`fail_on`** — The configured gate types whose findings must reject the artifact.
- **Finding fingerprint** — A stable identifier derived from relevant finding details, used to scope a waiver precisely.
- **False positive** — A scanner reports a problem that is not actually exploitable or sensitive.
- **False negative** — A scanner misses a real problem. This is why passing the gates is a floor rather than proof of security.
- **Waiver** — A signed, scoped, expiring acceptance of a specific security finding without disabling the scanner.
- **Waiver attestation** — Separately signed evidence showing that every otherwise-blocking finding has a valid waiver.
- **Scope drift** — The artifact, finding, policy, or run changes after approval, making the old waiver or signature inapplicable.
- **Supply-chain integrity** — Confidence that source, dependencies, build image, tools, artifact, evidence, and release inputs were not silently substituted.
- **Image digest pinning** — Referring to a container image by immutable SHA-256 digest instead of a mutable tag such as `latest`.
- **Reproducible build** — Repeating the same declared inputs produces independently verifiable equivalent output.
- **Provenance** — Evidence describing where an artifact came from, which inputs and tools produced it, and under what policy.

## Scenario-security terms

- **Coverage gate** — Ensures every declared behavior has at least one development scenario and scenarios do not refer to undeclared behaviors.
- **Orphan scenario** — A scenario referring to a behavior ID that was never declared.
- **Mutation validation** — Alters expected observations to confirm an assertion can actually fail.
- **Inert or tautological assertion** — A check that passes almost anything, such as asking whether output contains an empty string.
- **Sharpness battery** — Multiple near-miss observation mutations used to test whether assertions discriminate good results from subtly wrong ones.
- **Property/fuzz scenario** — Generates many seeded inputs and checks a general invariant rather than one fixed example.
- **Invariant** — A property that must remain true, such as determinism, idempotence, round-trip preservation, or no lost updates.
- **Deterministic seed** — A recorded random seed that makes generated cases reproducible.
- **Concurrency invariant** — A requirement evaluated under parallel execution, such as no lost updates or no hangs.
- **Probabilistic detection** — Repeated concurrency or fuzz testing increases the chance of finding bugs but cannot prove their absence.
- **Decorrelated critic** — A separate reviewer intended to have different blind spots from the builder and scenario author.
- **Model diversity** — Using different providers or models for authoring, building, and critique to reduce correlated mistakes.

## Builder and orchestrator confinement

- **Builder confinement** — Restricting the builder CLI to an explicitly permitted tool surface, ideally excluding web access, MCP, subagents, and arbitrary shell tools.
- **Adapter boundary** — The point where Dark Factory invokes a builder executable and applies confinement policy.
- **Structural confinement** — Security inferred from an adapter's implementation structure, such as a direct HTTP adapter having no agentic tool system.
- **Probe-verified confinement** — Attempting a forbidden tool operation and verifying that the tool is unavailable.
- **MCP surface** — Model Context Protocol tools that may expose files, applications, network services, or other capabilities to an AI process.
- **Orchestrator** — The outer AI session operating Dark Factory. The skill can sandbox the builder but cannot sandbox the session executing the skill.
- **Orchestrator lockdown** — Harness-level restrictions on the outer session's tools, MCP configuration, hooks, and OS access.
- **PreToolUse hook** — A policy hook that examines and permits or denies a tool call before execution.
- **Strict MCP configuration** — Starting the orchestrator with only explicitly configured MCP servers rather than accepting ambient or injected servers.

## Shipping and operational safety

- **Shipping boundary** — The point where a qualified artifact would be merged, deployed, migrated, or otherwise affect a real system.
- **Reversible action** — A ship action the OPERATOR declares reversible (by supplying a `rollback` argv). It is an operator ASSERTION, not verified: `reversible` is a required per-action flag and `rollback` is optional, so the runner trusts the declaration. The signed-release cover-list is the real authority for what runs unattended.
- **Irreversible action** — An operation that cannot reliably be undone; it requires stronger assurance and signed release approval.
- **Interlock** — A policy condition preventing a high-risk action until required controls are satisfied.
- **Release attestation** — K-of-N signed authorization bound to a specific run, artifact, and set of shipping actions.
- **Rollback-in-reverse** — If a later shipping action fails, previously completed reversible actions are undone in reverse order.
- **Unknown outcome** — A crash occurred after an external operation may have started but before its result was durably recorded.
- **Crash idempotence** — Resume logic avoids silently repeating an operation that may already have succeeded or incurred cost.
- **Reserve before dispatch** — Persisting the intent and budget reservation before initiating a model or shipping call.
- **Reconciliation** — Inspecting external state and deciding whether to retry, accepting duplicate risk, or abort.
- **Residual risk** — A known risk that remains after implemented controls and is documented so qualification is not overstated.
